    - Plan Implementacji
        
        ### **Faza 1: Backend (API & Baza Danych)**
        
        Celem jest przygotowanie solidnych fundamentów do przechowywania i zarządzania danymi.
        
        **Krok 1: Definicja Modeli w Bazie Danych**
        
        Potrzebujemy trzech głównych tabel: `projects`, `spaces` i artifacts.
        
        1. **Tabela `spaces`:**
            - id (UUID, Primary Key)
            - name (String) - Nazwa Space'a
            - `canvas_data` (JSONB) - Przechowuje całą strukturę kanwy, w tym zdefiniowane `Workspaces` (ramki/strefy) i ich zawartość.
            - created_at, updated_at
        2. **Tabela `projects`:**
            - id (UUID, Primary Key)
            - name (String) - Nazwa projektu
            - status (String, np. 'IDEA', 'IN_PROGRESS', 'COMPLETED')
            - `notion_url` (String) - Link do "Source of Truth"
            - `space_id` (UUID, Foreign Key do spaces.id, **może być NULL**) - Kluczowe dla naszego modelu.
            - `key_resources` (JSONB) - Do przechowywania linków do Figmy, GitHub, itp.
            - created_at, updated_at
        3. **Tabela artifacts (aktualizacja):**
            - Upewnić się, że project_id jest poprawnie powiązany z tabelą `projects`.
        
        **Krok 2: Implementacja Endpointów API**
        
        Należy stworzyć zestaw endpointów REST API do zarządzania projektami.
        
        1. **`POST /projects` (Tworzenie Projektu):**
            - Przyjmuje dane: name, `notion_url`, status.
            - Ma logikę wyboru:
                - `createNewSpace: true` -> tworzy nowy `Space` i od razu go przypisuje.
                - `existingSpaceId: "uuid"` -> przypisuje istniejący `Space` do projektu.
            - Zwraca nowo utworzony obiekt projektu.
        2. **`GET /projects` (Lista Projektów - dla Overview):**
            - Zwraca listę wszystkich projektów.
            - Dla każdego projektu agreguje kluczowe dane:
                - Nazwę, status.
                - Listę `Workspaces` (dynamicznie odczytaną z `canvas_data` powiązanego `Space`).
                - Liczbę artefaktów i ich podsumowanie statusów (np. `total: 6, approved: 5, in_review: 1`).
        3. **GET /projects/{id} (Szczegóły Projektu - dla Summary):**
            - Zwraca pełne dane jednego projektu.
            - Zawiera:
                - Podstawowe dane (name, status, `notion_url`, `key_resources`).
                - Listę `Workspaces` (z `canvas_data`).
                - Pełną listę powiązanych `Artifacts`.
                - Historię ostatnich aktywności (może wymagać osobnej tabeli `events` lub logiki).
        4. **PUT /projects/{id} (Aktualizacja Projektu):**
            - Pozwala na zmianę nazwy, statusu, linków, etc.
        5. **`GET /spaces?unassigned=true` (Lista wolnych Spaces):**
            - Zwraca listę `Space`'ów, które nie są przypisane do żadnego projektu (`space_id` w tabeli `projects` jest NULL). Niezbędne dla formularza.
        
        ---
        
        ### **Faza 2: Frontend (Komponenty & Widoki)**
        
        Implementacja interfejsu użytkownika na podstawie przygotowanego API.
        
        **Krok 1: Widok `Projects Overview`**
        
        1. **Strona Główna (`/projects`):**
            - Pobiera dane z `GET /projects`.
            - Implementuje filtry i sortowanie po stronie klienta lub przez parametry API.
        2. **Komponent `ProjectCard`:**
            - Renderuje pojedynczą kartę projektu.
            - Wyświetla nazwę, tagi `Workspaces`, podsumowanie artefaktów i status.
            - Zawiera przyciski: `[Open Space]`, `[[Szczegóły >]]`.
        3. **Przycisk `[+ Nowy Projekt]`:**
            - Otwiera modal do tworzenia projektu.
        
        **Krok 2: Widok `Create/Edit Project Modal`**
        
        1. **Komponent `ProjectFormModal`:**
            - Zawiera pola: Nazwa, Link do Notion, Status.
            - Implementuje logikę wyboru `(•) Utwórz nowy Space` vs `( ) Przypisz istniejący`.
            - Jeśli wybrano "Przypisz", pobiera listę wolnych `Space`'ów z `GET /spaces?unassigned=true`.
            - Wysyła dane do `POST /projects` (lub PUT /projects/{id} w trybie edycji).
        
        **Krok 3: Widok Project Summary**
        
        1. **Strona Szczegółów (/projects/{id}):**
            - Pobiera dane z GET /projects/{id}.
        2. **Komponent `KeyResources`:**
            - Wyświetla listę linków z `key_resources`. Umożliwia dodawanie/usuwanie.
        3. **Komponent `WorkspacesChips`:**
            - Renderuje dynamicznie pobrane `Workspaces` jako klikalne tagi/filtry.
        4. **Komponent `RecentActivity`:**
            - Wyświetla listę ostatnich zdarzeń.
        5. **Komponent `ArtifactsList`:**
            - Wyświetla listę artefaktów.
            - Implementuje logikę filtrowania (reaguje na kliknięcia w `WorkspacesChips`) i sortowania.

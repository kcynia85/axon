## 2. Entities & Domain Vocabulary
### Core Entities
*   **Project (Agregat):** The central container for user work. Groups Artifacts, Memories, and Tasks.
    *   *CodeName:* `Project` (Python Model), `Project` (Prisma/SQL).
*   **Agent (Entity):** An autonomous actor with a specific Role (Manager, Researcher, Builder).
    *   *CodeName:* `AgentConfig`, `AgentInstance`.
*   **Artifact (Entity):** A tangible output generated by an Agent (e.g., Markdown file, Code snippet, JSON).
    *   *CodeName:* `Artifact`.
*   **Memory (Value Object):** A semantic fact or knowledge chunk stored in the Vector DB (Path A - Wisdom).
    *   *CodeName:* `MemoryItem`.
*   **Asset (Entity):** A structured knowledge tool (Template, SOP, Checklist) preserved in full content (Path B - Assets).
    *   *CodeName:* `Asset` (SQL + Description Embedding).
*   **ChatSession (Entity):** A continuous interaction stream between User and Agent.
    *   *CodeName:* `ChatSession`.

### Key Actions
*   **Orchestrate:** The act of the Manager agent delegating a task to a sub-agent.
*   **Grounding:** The process of verifying AI responses against retrieved knowledge chunks.
*   **Fallback:** Switching from the primary LLM (Gemini) to a backup (GPT-4/Claude) upon failure.
*   **Asset Retrieval:** The specific act of fetching a full template/SOP by slug (`get_asset`) vs searching for chunks.

## 3. System Rules (Invariants)
> **Note:** These are immutable system rules that must be adhered to throughout the implementation.

1.  **Trustworthy Attribution:** Every answer derived from RAG MUST include inline citations (`[1]`, `[Source]`) pointing to specific `Memory` or `Artifact` chunks.
2.  **Project Isolation:** Users can ONLY access Projects and Artifacts where they are the owner (enforced via Supabase RLS).
3.  **Idempotent Execution:** Agent actions (especially side-effects like sending emails or writing files) MUST be idempotent to support durable execution (retries).
4.  **Contract-First UI:** All Generative UI components MUST have a pre-defined Zod schema; the backend MUST NOT stream invalid JSON structures to the frontend.
5.  **Fallback Resilience:** If the primary model (Gemini) returns a 429 or 5xx error, the system MUST automatically retry with a fallback model defined in the configuration.
6.  **Asset Integrity:** Templates and SOPs (Assets) must be retrieved in their entirety (not chunked) to preserve their structure for Agent use.
7.  **Continuous Documentation:** Every new feature MUST be accompanied by Technical Documentation (JSDoc/Docstring) and User Documentation updates (README/Guides) before completion.
8.  **Regression Testing:** Unit/Integration tests MUST be executed and pass after every significant change to ensure no regressions.

## 4. System Architecture & Constraints
*   **Architecture Pattern:** Modular Monolith with Vertical Slice Architecture (Backend).
*   **Backend Stack:** Python 3.10+, FastAPI, Google GenAI SDK (ADK).
*   **Frontend Stack:** Next.js 14/15 (App Router), Tailwind CSS, Shadcn/UI, Vercel AI SDK (streamUI).
*   **Database:** Supabase (PostgreSQL for relational, pgvector for semantic).
*   **Knowledge Pipeline (ETL):** Router-based ingestion splitting content into "Wisdom" (Chunks) and "Assets" (Full Text).
*   **State Management:**
    *   *Backend:* Stateless REST API (session state in DB).
    *   *Frontend:* React Server Components + Streaming (Suspense).
*   **Validation:** Pydantic (Backend), Zod (Frontend).
*   **Memory Check:** `knowledge/memory.md` reviewed? Yes.

## 5. Data Tier / DB Requirements
### SQL Tables (Relational)
- [x] `projects`: `{ id: uuid, name: text, status: enum, owner_id: uuid, domain: text, created_at: timestamp }`
- [x] `artifacts`: `{ id: uuid, project_id: uuid, type: enum, content: text, metadata: jsonb, version: int }`
- [x] `chat_sessions`: `{ id: uuid, project_id: uuid, agent_role: enum, history: jsonb }`
- [x] `assets`: `{ id: uuid, slug: text (unique), title: text, content: text, type: text, domain: text, metadata: jsonb, description_embedding: vector(768) }`

### Vector Collections (pgvector via `vecs`)
- [x] `knowledge_base`: Chunks of documents (Wisdom/Articles) with embeddings.
- [x] `memories`: Extracted facts with embeddings.

### JSONB Usage (NoSQL Hybrid)
- [x] **Artifact Metadata:** Flexible tags, source info, and layout configs.
- [ ] **Agent Logs:** Detailed execution traces (telemetry) stored in a `logs` column/table for audit, indexed via GIN.
- [x] **Asset Metadata:** Source path, versioning info.

### Performance Constraints
- [ ] Use GIN indexes on JSONB columns for efficient filtering by tags/metadata.
- [ ] Vector search must use HNSW index for sub-second retrieval.
- [x] Asset retrieval by slug must be sub-50ms (Indexed).

## 6. Sync vs Async Operations
### Synchronous (Direct Response)
*   Creating a new Project.
*   Fetching Project details/dashboard.
*   Retrieving a specific Asset by slug (`get_asset`).

### Asynchronous (Background/Streaming)
*   **Agent Reasoning Loop:** The "Think -> Tool -> Act" cycle runs asynchronously, streaming partial results via SSE.
*   **Knowledge Ingestion (ETL):** The `ingest_knowledge.py` script runs as a background job to process the `docs/axon-context` folder.
*   **Durable Workflows:** Long-running research tasks managed by the workflow engine (Inngest).

## 7. Caching Rules
*   **Semantic Cache:** Cache LLM responses for identical queries (cosine similarity > 0.95) using Redis or Vector store check.
*   **Asset Cache:** Frequently used templates (e.g., PRD Template) should be cached in memory/Redis.

# PRD Guide (Single-Module)

### **1. Wizja produktu i cele biznesowe**

<aside>

To absolutna baza. Zanim opiszesz wymagania, musisz wiedzieć:

- *dlaczego* produkt powstaje,
- jaki problem rozwiązuje,
- jaki cel ma dowieźć (np. zwiększenie aktywacji, redukcja kosztów, nowe źródło przychodów).

Bez tego PRD będzie tylko listą funkcji bez kierunku.

</aside>

### **2. Badania użytkowników i insighty**

<aside>

PRD powinno wynikać z realnych potrzeb, nie przypuszczeń. Najczęściej używa się:

- wywiadów,
- badań ilościowych,
- mapy bólu i potrzeb,
- person,
- obserwacji zachowań w produkcie,
- feedbacku od supportu i sprzedaży.

To w PRD powinno wybrzmieć jako: *problem użytkownika, kontekst, zachowanie, potrzeba*.

</aside>

### **3. Business logic + constraints**

<aside>

Czyli:

- ograniczenia technologiczne,
- ograniczenia prawne,
- ograniczenia związane z budżetem,
- zależności z innymi zespołami i systemami,
- wymagania bezpieczeństwa.

PRD musi uwzględniać realne warunki, żeby nie tworzyć funkcji, których nie da się zbudować.

</aside>

### **4. UX: user flow, journey, scenariusze**

<aside>

To jest klucz, bo PRD opisuje *jak użytkownik ma osiągać cel w produkcie*.

W praktyce:

- user flow (krok po kroku),
- scenariusze użycia,
- definicja edge cases,
- propozycja interakcji i layoutu (czasem szkice).

PRD nie musi być UI, ale powinien tłumaczyć przepływ zachowań.

</aside>

### **5. Dane i metryki, które określają sukces**

<aside>

To jest sekcja często pomijana, a kluczowa:

- jakie metryki zmierzą, czy funkcja działa?
- co będzie KPI dla tego PRD?
- kiedy uznamy, że funkcjonalność jest “done right”, nie tylko “done”?
</aside>

### **6. Backlog wysokopoziomowy**

<aside>

Czyli:

- lista funkcji / elementów do zbudowania,
- opis ich wartości,
- powiązanie z celami,
- priorytety.
</aside>

### **7. Benchmarki i inspiracje**

<aside>

PRD staje się mocniejsze, jeśli pokazuje przykłady:

- jak rozwiązują to inni,
- jakie standardy UX istnieją,
- jakie są best practices.
</aside>

## ✔️ Podsumowanie w jednym zdaniu

<aside>

**PRD powinien powstać na bazie: wizji produktu, danych o użytkownikach, ograniczeń technologicznych/prawnych, user flow, KPI oraz wysokopoziomowego backlogu funkcji.**

</aside>
